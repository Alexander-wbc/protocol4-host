# Protocol 4 单窗口 UDP 演示平台

一个基于 Python 的 **协议四（Protocol 4）** 图形化演示程序。  
程序采用 **单窗口界面**，但内部同时运行两个真实的 **UDP 节点 A / B**，用于在一台电脑上完成协议四的发送、接收、确认、超时重传与错误处理演示。

---

## 一、协议四简介

协议四通常出现在计算机网络课程中，用于说明**数据链路层可靠传输**的基本思想。它可以看作一种 **1 比特滑动窗口协议**，本质上属于**停等式可靠传输协议**。

在协议四中：

- 发送方一次只允许发送一个未确认的数据帧
- 每发送一个数据帧后，必须等待接收方返回 ACK
- 如果在规定时间内没有收到 ACK，就认为该帧或 ACK 在传输过程中丢失或损坏，于是触发**超时重传**
- 接收方通过帧序号判断当前收到的是**新帧**还是**重复帧**

由于协议四的窗口大小为 1，因此序号空间只需要 1 bit，即：

- `0`
- `1`

发送方和接收方都只需要在 `0` 和 `1` 之间切换状态即可。

### 协议四解决的问题

协议四主要解决以下几类典型问题：

1. **数据帧丢失**  
   发送方发出的数据帧没有成功到达接收方。  
   处理方式：发送方超时后重传。

2. **ACK 丢失**  
   接收方已经正确收到数据并返回 ACK，但 ACK 在途中丢失。  
   处理方式：发送方超时后重传原数据帧；接收方通过序号识别重复帧，不重复交付，只重发 ACK。

3. **帧损坏**  
   数据帧或 ACK 在传输过程中被破坏。  
   处理方式：校验失败后丢弃，发送方因收不到有效 ACK 而超时重传。

4. **重复帧**  
   由于重传机制，接收方可能再次收到已经正确接收过的旧帧。  
   处理方式：接收方识别为重复帧，不重复交付，仅重发 ACK。

### 协议四的核心思想

协议四最核心的机制有三个：

#### 1. 序号机制
每个数据帧都带有一个序号 `seq`，其取值只有 `0` 和 `1`。  
接收方通过比较收到帧的序号与当前期望序号 `expected_seq`，判断该帧是否是新帧。

#### 2. ACK 确认机制
接收方在正确接收到一个新数据帧后，会向发送方返回一个 ACK 帧，表示该数据已经被成功接收。

#### 3. 超时重传机制
如果发送方在超时时间内没有收到有效 ACK，就会自动重传上一个数据帧，从而保证通信可靠性。

### 协议四为什么重要

协议四虽然结构简单，但它体现了可靠传输协议最关键的思想：

- 发送确认
- 超时重传
- 序号判重
- 防止重复交付

因此它是学习更复杂滑动窗口协议（如回退 N 帧、选择重传）的基础。

## 二、项目简介

本项目使用 Python `tkinter` 编写图形界面，使用 `socket` 实现 UDP 通信，内部在本机回环地址上创建两个节点：

- 节点 A：`127.0.0.1:9000`
- 节点 B：`127.0.0.1:9001`

两个节点之间通过真实 UDP socket 通信，但所有过程都集中显示在同一个窗口中，便于课程展示与实验答辩。

本项目实现了协议四的核心机制：

- 1 bit 序号 `0/1`
- 停等发送
- ACK 确认
- 超时重传
- 重复帧识别
- CRC32 校验
- 确定性错误注入演示

---

## 三、主要功能

### 1. 单窗口双节点显示
程序只打开一个 GUI 窗口，但内部同时运行两个协议四节点：

- A 节点
- B 节点

支持：

- A → B 发送消息
- B → A 发送消息

### 2. 核心状态显示
每个节点界面中只保留三个核心状态：

- `send_seq`：下一次发送的数据帧序号
- `expected_seq`：当前期望接收的数据帧序号
- `ACK状态灯`
  - 绿色：空闲
  - 红色：正在等待 ACK

### 3. 总日志显示
日志窗口会显示完整协议过程，例如：

- DATA 发送
- ACK 发送
- DATA 接收
- ACK 接收
- 校验失败
- 超时重传
- 重复帧识别
- 错误注入信息

### 4. 错误演示模式
本项目支持几种确定性错误注入模式，便于稳定演示协议四的错误恢复能力：

- `normal`
- `drop_first_data`
- `drop_first_ack`
- `corrupt_first_data`
- `corrupt_first_ack`
